# Consistency invariants inventory

Read-only inventory of every implicit invariant found in the codebase: every
place where two or more pieces of state must agree, or where a single stored
value must stay internally consistent. This is the ground map for a future
deterministic, rule-based Consistency Checker (pure code, no AI, read-only,
pass/fail plus a log).

Every bug in this class has the same shape: two pieces of state that are meant
to agree drift apart, and a human only catches it by eyeballing the Final
Report. Three have already been fixed (a "Remove all" that left a stale
holisticBand inflating the score, commit 7da9b1e; the verdict/comment
`conclusionMismatch` guard; Final Report grouping lines by official ref not
stored tag, commit 6ddeb7c). The rows below are the full set that remain.

Investigation only. No application code was changed to produce this document.

## How to read the "Guarded?" column

- **yes** - a runtime guard enforces the invariant at the write or compile site; the named guard is the thing to reuse or lean on.
- **partial** - a guard exists but does not cover every path, or only advises rather than enforces, or holds by design in one direction only.
- **no** - nothing enforces it; the two values can drift with no signal. These are the prime candidates for the checker.

## Open questions

1. INV-08 (stored band vs live matrix after a scale edit): no reader that displays `holisticBand.band`/`.totalPct` directly (rather than re-deriving via `apsrMatrixResult`) was confirmed during this pass. Whether the stale snapshot ever reaches a user-visible surface, or is purely internal, needs a targeted sweep of every `holisticBand.band`/`.totalPct` read before deciding if INV-08 is a real display bug or only a latent one.
2. INV-13 (gate mixes band provenances): a single gate average can blend a matrix-derived band and an evidence-capped band. This may be intended (an item is scored by whichever model has data), but it means the "min Band 3" gate is not guaranteed to compare like with like. Confirm with the scoring owner whether a single provenance per gate is required.
3. INV-14 vs INV-01: `clearSpecificLines` clears both `apsrMatrix` and `holisticBand.matrixScores`, but `setApsrMatrix` can leave the live `apsrMatrix` ahead of the saved band indefinitely. Is a live matrix that diverges from the saved band something the checker should flag, or is it expected mid-edit state that should be ignored until the next save?
4. INV-16 (`Finding.overdue` inert): the stored field is dead (always `false`) and every consumer uses the derived `isFindingOverdue`. The checker could either flag the stored/derived disagreement or the field could be treated as deprecated. Decide whether to keep the field at all.
5. INV-10 sentinel robustness: the "Not assessed" state travels as a free-text prefix rather than a structural flag. A structural `assessed: false` on the APSR dimension would make the invariant enforceable rather than string-coupled. Flagging as a design question, not a decision for this pass.
